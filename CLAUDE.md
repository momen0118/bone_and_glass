# 骨と硝子の店 — Os et Vitrum

19世紀末の標本商になり、亡骸や鉱石を仕入れ、標本に仕立て、硝子棚に並べて売る経営シミュレーション。React単一コンポーネントのブラウザゲーム。

## 役割分担(重要)

- ゲームの仕様・数値バランス・文言・世界観の決定権は claude.ai 側の会話(設計者)にある。
- このリポジトリで作業するClaude(あなた)は実装担当。
- **指示にない仕様変更・バランス調整・文言変更・機能追加を勝手に行わないこと。**
  気づいた改善案は実装せず、コミットメッセージや報告に「提案」として書き添える。
- リファクタリングは挙動を一切変えない範囲でのみ可。

## 設計思想

- 参考: レミュオールの錬金術師(仕入→加工→販売のループ、返済目標)、ザ・コンビニ(陳列)、まんぷくマルシェ(レシピ発見・図鑑)、コーヒーインク2(値付け・客の予算)、だんじょん商店会(セリフの立った客、交渉)。
- 「発見」を大切にする: レシピ・銘板セット・通り名は、条件を明示せず遊びの中で見つかる。ネタバレになるUI表示を足さない。
- 経験値バー等の生々しい数値露出は避ける(熟練Lvの表示は可、進捗量の表示は不可)。
- プレイヤーは一人(設計者本人)。モバイル(縦画面・親指操作)を第一に。

## 世界観・文体

- 薄暗い店内、銅版画、真鍮、羊皮紙、硝子瓶。配色はコード内の定数 `C` を使う(新しい色を勝手に増やさない)。
- 地の文は短く静かに。「……」や体言止めを効かせる。説明口調にしない。
- 客のセリフの声色:
  - 学生: 現代寄りの素直な敬語。金がない。
  - 老学者: 老練、断定調。「ふむ」「〜のでな」。
  - 好事家: 芝居がかった紳士。「〜かね」。
  - 貴婦人: 柔らかい。「〜ですの」「〜だわ」。
  - 外套の蒐集家: 口数少なく、間に「……」。丁寧だが得体が知れない。
- 新規セリフが必要な場合も、まず設計者に確認するのが原則。

## 技術的約束

- React単一コンポーネント構成を尊重する。分割はデータ定義(素材・標本・レシピ等)をファイルに切り出す程度まで。状態管理ライブラリ等は導入しない。
- 依存は最小限。UIフレームワークを追加しない(スタイルはインラインstyle)。
- セーブ互換: `SAVE_KEY`(bone_glass_save_v1)と `IMG_KEY` は変更しない。状態に項目を足すときは必ず `migrate()` にデフォルト値を追加し、旧セーブが壊れないようにする。
- ストレージは両対応ラッパー経由: `window.storage`(Claudeアーティファクト環境)があればそれを、なければ `localStorage` を同じasync API形状(get/set/deleteで `{value}` を返す)で使う。ゲームコードはラッパーだけを呼ぶ。
- 画像は「あれば使う、なければ劣化なしにフォールバック」が原則。絵文字フォールバックを削除しない。

## 画像の命名規則

リポジトリにこの名前で置かれた画像は自動で読み込む。拡張子は `.png` と `.jpg` の両方に対応し、png→jpgの順で探す(同名が両方あればpng優先)。無ければ画廊(ゲーム内アップロード)→絵文字の順にフォールバック。

```
img/shop.(png|jpg)                     店内観(タイトル背景)
img/portraits/gakusei.(png|jpg)        学生
img/portraits/gakusha.(png|jpg)        老学者
img/portraits/koujika.(png|jpg)        好事家
img/portraits/kifujin.(png|jpg)        貴婦人
img/portraits/collector.(png|jpg)      外套の蒐集家
img/specimens/{標本ID}.(png|jpg)       標本(IDはコード内SPECIMENSのキー。例 s_torikotsu.png)
```

## 動作確認(実装後に必ず)

- 新規開始 → 1日ループ(朝→工房→陳列→夜→翌朝)が回る
- 「続きから」で旧バージョンのセーブが読める(migrate)
- 画像が1枚も無い状態で表示が破綻しない
- モバイル幅(〜390px)でレイアウトが崩れない
